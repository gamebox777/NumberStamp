# TypeScript (TSX) への移行検討

## 結論
**現状維持 (JavaScript/JSX)** が推奨される。

現状の「AIのアシストを受けながら個人開発をする」という状況において、機能追加やスピードを優先する場合、JavaScriptのまま進めるのが合理的。

## 比較：AI開発におけるメリット・デメリット

| 項目 | TypeScript (TSX) の場合 | JavaScript (JSX) の場合 (現状) |
| :--- | :--- | :--- |
| **信頼性** | ⭕ **AIの「嘘」が減る**<br>AIが適当なコード（存在しない関数やプロパティ）を書くと、コンパイルエラーが出るため、**実行前に**気づける。 | ❌ **実行するまでミスに気づかない**<br>実行時エラーになるまで気づけない場合がある。 |
| **修正の手間** | ❌ **「型パズル」が発生する**<br>機能は実装できているのに、「型が合わない」というエラー解消のためにAIと数往復する無駄な時間が発生する。 | ⭕ **「とりあえず動く」が早い**<br>型定義を無視して、動くコードを最速で作れる。プロトタイピングやアイデア実装には最適。 |
| **コンテキスト** | ⭕ **AIへのヒントが増える**<br>型定義自体がドキュメント代わりになるため、AIが既存コードの意図を汲み取りやすくなる。 | 🔺 **推測に頼る**<br>AIは変数名や使い方から型を推測するため、複雑なロジックでは誤解が生じることがある。 |
| **移行コスト** | ❌ **今は足枷になる**<br>既存の全ファイルを書き換える必要があり、一時的に機能開発がストップする。 | ⭕ **今の資産をそのまま使える**<br>移行作業ゼロで、今やりたい昨日の実装に集中できる。 |

## 今後の判断基準
以下のような状況になったら移行を検討する：
- 機能追加よりも、**既存機能の改修（リファクタリング）** がメインになってきた時。
- 「最近、単純なバグ（nullエラーなど）が多いな」と感じ始めた時。
